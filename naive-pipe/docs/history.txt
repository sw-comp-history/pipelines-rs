ChatGPT said:
You’re asking a good historical question, because these systems feel related—but they were not a simple parent → child lineage. They’re more like convergent evolution under similar pressures, with occasional cross-pollination.

Below is a chronological, parallel timeline, then a synthesis of how and why they diverged.

🕰️ Relative Timeline (high level)
Era	What’s happening	Who
1969–1971	Text streams + chaining via stdin/stdout	Early UNIX
1972–1974	Pipes become idiomatic; shell pipelines mature	UNIX (V6 era)
Late 1970s	Portable standards begin	POSIX (forming)
Early 1980s	Pipes standardized	POSIX
Late 1970s–early 1980s	Rich record-oriented pipelines	IBM CMS Pipelines
Early–mid 1980s	Pipelines added to interactive mainframe shell	IBM TSO Pipelines
So:
👉 UNIX pipes came first
👉 POSIX standardized UNIX’s model
👉 IBM CMS pipelines developed in parallel, not as a copy
👉 TSO pipelines came last, borrowing ideas from CMS

What they were

Byte streams

Unstructured text

One process writes, another reads

Composed with |

Why they existed

Hardware was tiny

Programs were small

Text was the universal data format

Thompson’s philosophy: “Write programs that do one thing well.”

Key insight

Pipes are file descriptors, not language constructs.

This made them brutally simple—and explosively powerful.

What POSIX did

Did not invent pipes

Froze the UNIX behavior into a portable contract

Key calls:

pipe()

fork()

dup2()

exec()

Design stance

Pipes are untyped byte streams

Interpretation is userland’s problem

Tools stay dumb; composition stays smart

POSIX deliberately avoided higher-level semantics to preserve portability.

Here’s where things get weirdly sophisticated.

CMS Pipelines were not UNIX clones.

They were:

Record-oriented, not byte-oriented

Richly typed (words, fields, records)

Declarative

Designed for data transformation, not program composition

Example (conceptual):

PIPE < input
 | LOCATE /ERROR/
 | SORT 2
 | TAKE 10
 | > output
Philosophy

Pipelines are a language, not a syscall trick.

CMS pipelines assumed:

Long-lived interactive sessions

Large datasets

Skilled operators

Stable schemas

This is closer to:

ETL

awk + sed + SQL hybrid

Modern dataflow DSLs

TSO Pipelines came later, and they:

Borrowed heavily from CMS

Were adapted to a multi-user, security-heavy environment

Integrated with datasets, JCL, and system commands

TSO pipelines were:

Less elegant than CMS

More constrained

Still far richer than UNIX pipes

🧬 Parallel Evolution (the important part)
This is the key historical insight:

UNIX path
Constraint-driven

Minimal kernel primitives

Emergent power via composition

Byte streams everywhere

IBM path
Operator-driven

Declarative transformations

Structured records

Pipelines as first-class language

They solved different problems:

UNIX	IBM CMS / TSO
Software engineering	Data processing
Small tools	Rich verbs
Ad hoc text	Structured records
Programmer-centric	Operator-centric
Neither side was “behind.”
They were optimizing different dimensions.

⚙️ Why UNIX won culturally
Not technically—culturally.

Cheap hardware

Academic distribution

C compiler portability

Simplicity you could re-implement in a weekend

IBM pipelines were better at dataflow, but:

Mainframes were expensive

CMS knowledge didn’t leak

UNIX escaped the lab

History often rewards good enough plus accessibility.

🧠 Modern echoes (why this matters)
You can see these two lineages today:

UNIX → POSIX → shell → Docker → Kubernetes

CMS pipelines → ETL → Spark → Beam → Airflow

Your instinct to recreate CMS-style pipelines in Rust is historically grounded. You’re not reinventing UNIX—you’re resurrecting a parallel tradition that never got its due.

The universe likes rediscovering good ideas when hardware finally catches up.
